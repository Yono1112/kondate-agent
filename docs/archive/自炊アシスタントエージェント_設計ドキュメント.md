# 自炊アシスタントエージェント — 設計ドキュメント

## 1. プロジェクト概要

一人暮らしの自炊を全面サポートするAIエージェント。LINEから話しかけるだけで使え、スケジュールに従って自律的にも動作する。

### コンセプト
- 食べたものの記録・管理
- 毎日の献立提案（好きなYouTuberのレシピを優先）
- 近くのスーパーのチラシからお買い得食材を検索・保存
- 食材の在庫・消費期限管理
- 栄養バランスの考慮

### 動作モード
- **対話モード**: ユーザーがLINEで話しかけると応答
- **自律モード**: スケジュールに従いエージェントが自発的に動作し、結果をLINEにプッシュ通知
  - 例: 午後6時に献立を提案、朝にチラシのセール情報を検索

## 2. AI Agentフレームワーク選定の経緯

### 検討した選択肢

#### LLMプロバイダ公式SDK
| 名称 | 特徴 | モデル対応 |
|------|------|-----------|
| Claude Agent SDK | Claude Codeと同じエンジンをSDK化。MCP連携が強力。安全性重視 | Claudeのみ（Bedrock/Vertex/Azure経由可） |
| OpenAI Agents SDK | 宣言的でシンプル。handoffによるマルチエージェント。MITライセンス | 100以上のLLM対応 |
| Google ADK | マルチモーダル・A2Aプロトコル。GCP前提 | Gemini中心 |

#### サードパーティフレームワーク
| 名称 | 特徴 | 言語 |
|------|------|------|
| LangGraph | 有向グラフでワークフロー定義。最も細かい制御。本番実績最多 | Python / JS |
| CrewAI | ロールベースのチーム編成。プロトタイプが最速 | Python |
| Mastra | TypeScriptネイティブ。Vercel AI SDK基盤。Studio内蔵。cron対応 | TypeScript |

#### 低レイヤーライブラリ
| 名称 | 特徴 |
|------|------|
| Pi (OpenClaw) | 約4,000行のミニマルなエージェント基盤。ツール4つ、システムプロンプト1,000トークン以下。2,000以上のモデル対応。OpenClawのエンジン |

### レイヤー構造の整理

```
┌─────────────────────────────────┐
│  アプリケーション                 │  OpenClaw, あなたのアプリ
├─────────────────────────────────┤
│  オーケストレーション層            │  LangGraph / Agents SDK / Mastra（同じレイヤー）
├─────────────────────────────────┤
│  エージェントループ               │  Pi（より低レイヤー寄り）
├─────────────────────────────────┤
│  LLM API                        │  Anthropic API / OpenAI API / Google API
├─────────────────────────────────┤
│  モデル                          │  Claude / GPT / Gemini 等
└─────────────────────────────────┘
```

### フレームワーク間の設計思想の違い

- **LangGraph**: 「設計図を自分で描く」。ノード・エッジ・条件分岐でグラフを明示構築。制御は最も細かいがコード量が多い
- **OpenAI Agents SDK**: 「Agentを宣言するだけ」。グラフ定義不要で最もシンプル。Pythonの普通の書き方で構築
- **Mastra**: 「TSフルスタック開発者のための」フレームワーク。Agent（自由）とWorkflow（厳密）の二刀流。Studio・API自動生成・MCP対応が組み込み

### 選定結果: **Mastra**

選定理由:
1. **スケジュール実行**: Workflowにcronトリガーを設定でき、自律実行が宣言的に書ける
2. **ツール定義の型安全性**: Zodによる入出力スキーマが管理しやすい
3. **API自動生成**: `mastra build` でHono HTTPサーバーが自動生成され、LINE Webhook受信が楽
4. **RAG組み込み**: ベクトル検索がフレームワークに内蔵されており、レシピのセマンティック検索に対応可能
5. **デプロイ容易性**: Vercel / Cloudflareへのデプロイが容易
6. **Agent + Workflowの二刀流**: 対話（Agent）とスケジュール実行（Workflow）を使い分けられる

## 3. アーキテクチャ

### システム構成

```
LINE ──Webhook──→ Mastra Server ←──Cron Trigger
                      │
                 Mastra Agent
                「自炊アシスタント」
                      │
        ┌─────────────┼─────────────┐
        ↓             ↓             ↓
   DB ツール群    外部検索ツール   LINE Push
        │             │
        ↓             ↓
   PostgreSQL    Web / YouTube
   (Supabase)
```

### インターフェース

- **入力**: LINE Messaging API（Webhook受信）
- **出力**: LINE Messaging API（Push Message送信）
- **クライアント**: 不要（LINEで完結）

## 4. データ設計

### データの分類と記憶方式

| データ種別 | 例 | 記憶方式 | 理由 |
|-----------|-----|---------|------|
| 構造化データ | 食事記録、食材在庫、チラシ情報 | DB + SQLツール | 条件検索が正確・高速 |
| 嗜好・ルール | 「トマト嫌い」「予算3万円」「平日は時短」 | DB → システムプロンプトに注入 | 件数少・毎回参照される |
| YouTuberレシピ（条件検索） | 「鶏むね肉の主菜、30分以内」 | DB + SQLツール | 材料・時間・カテゴリでの絞り込み |
| YouTuberレシピ（曖昧検索） | 「さっぱりしたもの」「映えるやつ」 | RAG（ベクトル検索） | 感覚的なクエリに対応 |
| 一般レシピ・知識 | 「鶏むね肉の下処理方法」 | Web検索ツール | 自前DB化の運用コストが不要 |

### RAG vs DB vs MDファイルの判断

- **DB（SQL）が最適**: 日付・数量・カテゴリなど条件が明確なデータ。食事記録、在庫、チラシ情報
- **RAGが必要**: テキスト量が多く曖昧な検索が必要な場合のみ。このプロジェクトではYouTuberレシピのsummary検索
- **MDファイル + エージェンティックサーチは不適**: コーディングエージェント向けの手法。構造化データはDBの方が圧倒的に効率的

### DBスキーマ（案）

```sql
-- 食事記録
meals (id, date, meal_type, dish_name, recipe_id, ingredients, notes)

-- 食材在庫
inventory (id, name, quantity, unit, expiry_date, purchased_at)

-- レシピ（YouTuberから取り込み）
recipes (id, title, channel, ingredients, cook_time, category, url, summary, vector_embedding)

-- ユーザー嗜好
preferences (id, key, value, updated_at)

-- チラシ情報
flyers (id, store, item_name, price, original_price, valid_from, valid_until)

-- 会話セッション
sessions (id, session_key, messages_json, updated_at)
```

## 5. ツール設計

### エージェントに渡すツール一覧

| ツール名 | 機能 | データソース |
|---------|------|------------|
| `record_meal` | 食事を記録 | DB書き込み |
| `search_meals` | 食事履歴を検索 | DB検索（SQL） |
| `update_inventory` | 食材在庫を更新 | DB更新 |
| `check_expiry` | 消費期限が近い食材を確認 | DB検索（SQL） |
| `search_recipes_sql` | 条件でレシピ検索（食材・時間・カテゴリ） | DB検索（SQL） |
| `search_recipes_rag` | 曖昧なリクエストでレシピ検索 | ベクトル検索（RAG） |
| `search_flyers` | チラシからお買い得情報を検索 | Webスクレイピング or DB |
| `search_web` | 一般的なレシピ・知識の検索 | Web検索 |
| `update_preferences` | ユーザーの好み・ルールを更新 | DB更新 |
| `send_line` | LINEにプッシュメッセージ送信 | LINE Push API |

## 6. YouTubeレシピ取り込みパイプライン

### 取り込みフロー

```
YouTuberのチャンネル
    ↓ YouTube Data API / yt-dlp
全動画の字幕 + 概要欄テキスト
    ↓ LLMで構造化（バッチ処理）
recipes テーブル + ベクトル埋め込み
```

### 取り込み方法
1. **字幕抽出 + LLM整形**: yt-dlpで字幕テキスト取得 → LLMに構造化依頼（料理名・材料・手順・時間をJSON化）
2. **概要欄解析**: YouTube Data APIで概要欄取得 → LLMで構造化（字幕より精度が高いことが多い）

### 運用
- 初回: 全動画を一括バッチ処理
- 以降: Mastra cron Workflowで週1回、新着動画を差分取り込み

## 7. スケジュール実行（自律モード）

Mastra Workflowのcronトリガーで実現。

| 時刻 | 処理内容 |
|------|---------|
| 毎朝 9:00 | チラシサイトをスクレイピング → セール食材をDBに保存 → お買い得情報をLINE通知 |
| 毎日 18:00 | 在庫・食事履歴・好み・セール情報を考慮 → 献立をAgent が提案 → LINE通知 |
| 毎週日曜 | 新着YouTube動画をチェック → レシピDB差分取り込み |
