# 自炊アシスタントエージェント — 設計ドキュメント

> **最終更新**: 2026-04-10

---

## 1. プロジェクト概要

一人暮らしの自炊を全面サポートするAIエージェント。LINEから話しかけるだけで使え、スケジュールに従って自律的にも動作する。

### コンセプト
- 食べたものの記録・管理
- 毎日の献立提案（好きなYouTuberのレシピを優先）
- 食材の在庫・消費期限管理
- スーパーのセール情報をもとにしたお買い得献立提案
- 栄養バランスの考慮

### 動作モード
- **対話モード**: ユーザーがLINEで話しかけると応答
- **自律モード**: スケジュールに従いエージェントが自発的に動作し、結果をLINEにプッシュ通知
  - 例: 毎日18時に在庫・セール情報を考慮した献立を提案

### 究極ビジョン（長期目標）
在庫管理 → 献立提案 → **不足食材の自動発注** まで一気通貫の完全自律オペレーション。
「冷蔵庫が空にならない生活」の実現。詳細は Phase 5 参照。

## 2. AIフレームワーク選定

### 選定結果: **Mastra（TypeScript）**

選定理由:
1. **スケジュール実行**: Workflow に cron トリガーを設定でき、自律実行が宣言的に書ける
2. **ツール定義の型安全性**: Zod による入出力スキーマが管理しやすい
3. **API自動生成**: `mastra build` で Hono HTTP サーバーが自動生成され、LINE Webhook 受信が楽
4. **RAG組み込み**: ベクトル検索がフレームワークに内蔵されており、レシピのセマンティック検索に対応可能
5. **Agent + Workflow の二刀流**: 対話（Agent）とスケジュール実行（Workflow）を使い分けられる

検討した選択肢（Claude Agent SDK / OpenAI Agents SDK / LangGraph / CrewAI 等）の比較は git 履歴の旧版を参照。

## 3. アーキテクチャ（現状）

### システム構成

```
LINE ──Webhook──→ Mastra Server ←──Cron Trigger（Phase 4〜で追加予定）
                      │
                 Mastra Agent
               「自炊アシスタント」
                      │
        ┌─────────────┼──────────────┐
        ↓             ↓              ↓
   DB ツール群    LINE Push       外部サービス
        │          ツール群      （Phase 3〜で追加）
        ↓
   LibSQL (mastra.db)
```

### インターフェース

- **入力**: LINE Messaging API（Webhook受信）/ レシート画像（LINE画像メッセージ）
- **出力**: LINE Messaging API（テキスト応答 or ボタンテンプレートメッセージ）
- **クライアント**: 不要（LINEで完結）

### メッセージ送信ルール

エージェントは **1ターン1メッセージ** を原則とする。

- **パターンA**: テキスト応答として返す（在庫確認・食事記録・期限チェック・雑談等）
- **パターンB**: `send-line-buttons` ツールで選択肢ボタンを送る（**献立提案時のみ**）。この場合はエージェントのテキスト応答は空にする

詳細は `docs/adr/2026-04-09-line-message-single-reply.md` 参照。

## 4. データ設計

### DBスキーマ（現在の実装）

```sql
-- 食材在庫
inventory (id, name, quantity REAL, unit, expiry_date, purchased_at, created_at, updated_at)

-- 食事記録
meals (id, date, meal_type, dish_name, ingredients TEXT, notes, created_at)
-- meal_type: 朝食 / 昼食 / 夕食 / 間食

-- ユーザー設定
preferences (id, key UNIQUE, value, updated_at)

-- 購入履歴（レシートOCRで自動登録）
purchases (id, store_name, item_name, price, quantity, unit, purchased_at, receipt_image_url, created_at)
```

### 追加予定テーブル（Phase 3〜）

```sql
-- レシピ（YouTubeから取り込み）
recipes (id, title, channel, ingredients TEXT, cook_time, category, url, summary, embedding BLOB)

-- 買い物リスト（Phase 4〜）
shopping_list (id, item_name, quantity, unit, priority, status, created_at)
```

### データの記憶方式

| データ種別 | 記憶方式 | 理由 |
|---|---|---|
| 食事記録・食材在庫・購入履歴 | DB（LibSQL） + SQLツール | 条件検索が正確・高速 |
| ユーザー嗜好・設定 | DB → システムプロンプトに注入 | 件数少・毎回参照される |
| YouTuberレシピ（条件検索） | DB（SQL） | 材料・時間・カテゴリでの絞り込み |
| YouTuberレシピ（曖昧検索） | RAG（ベクトル検索） | 「さっぱりしたもの」等の感覚的クエリ |
| 一般レシピ・知識 | Web検索 | 自前DB化の運用コストが不要 |

## 6. レシートOCR方式（現在の実装）

### 2段階解析フロー

```
レシート画像（LINE）
    ↓
Step 1: 店舗識別
  zhipuai/glm-5v-turbo で「この店舗名は？」
    ↓
Step 2: 店舗別プロンプトで詳細解析
  resolveStorePrompt(storeName) で店舗固有の知識を注入
    ↓
manage-inventory / purchases テーブルに自動登録
```

### 店舗別対応状況

| 店舗 | 対応状況 | 専用プロンプト内容 |
|---|---|---|
| サミット | **対応済み** | 部門コード表（1100=青果, 1300=精肉, 1520=加工食品, 1580=日配品, 4425=パン）、ブランド辞書（SB, ホクレン, ユウキ, 日清等）、複数購入行の解釈 |
| マルエツ | 未対応（フォールバック汎用） | — |
| ライフ | 未対応（フォールバック汎用） | — |

詳細な分析は `docs/research/2026-04-09-receipt-ocr-and-net-supermarket.md` 参照。

## 7. YouTubeレシピ取り込みパイプライン（Phase 3 で実装予定）

### 取り込みフロー

```
YouTuberのチャンネル
    ↓ YouTube Data API / yt-dlp
全動画の字幕 + 概要欄テキスト
    ↓ LLMで構造化（バッチ処理）
recipes テーブル + ベクトル埋め込み（Mastra RAG）
```

### 取り込み方法
1. **字幕抽出 + LLM整形**: yt-dlp で字幕テキスト取得 → LLM に構造化依頼（料理名・材料・手順・時間を JSON 化）
2. **概要欄解析**: YouTube Data API で概要欄取得 → LLM で構造化

### 運用
- 初回: 全動画を一括バッチ処理
- 以降: Mastra cron Workflow で週1回、新着動画を差分取り込み

## 8. スケジュール実行（自律モード）（Phase 4 で実装予定）

Mastra Workflow の cron トリガーで実現。

| 時刻 | 処理内容 |
|---|---|
| 毎日 18:00 | 在庫・食事履歴・設定・セール情報を考慮 → 献立を Agent が提案 → LINE Push |
| 毎週日曜 | 新着 YouTube 動画をチェック → レシピ DB 差分取り込み |
| 不定期（配達完了時） | Gmail 経由で配達完了メールを検知 → Computer Use で注文明細取得 → 在庫自動登録（Phase 5） |

### Phase 5: ネットスーパー連携・自動発注（究極ビジョン）

**目標**: エージェントが自律的にネットスーパーに発注する「完全自律購買」。

#### 実装ステップ

1. **Computer Use PoC（データ取得）**
   - Anthropic Computer Use reference impl（Docker / Xvfb + Firefox）をセットアップ
   - サミットネットスーパーにログインし、注文履歴を取得
   - 取得結果を `manage-inventory` に書き込み（LINE 承認ゲート付き）

2. **Gmail トリガー化**
   - 配達完了メール検知 → Mastra Workflow 起動 → Computer Use でデータ取得 → 在庫自動登録

3. **店舗拡大**
   - マルエツ・ライフ・Oisix 等（Computer Use の強みでコスト低）

4. **自動発注**
   - 在庫 + 献立 + 予算 → 不足品リスト生成
   - Computer Use でカート投入
   - LINE 承認ボタンで確認 → 承認後に購入確定
   - 安全策: 金額上限・週次予算・ホワイトリスト・高額品は必ず人間承認

#### なぜ Playwright MCP ではなく Computer Use か

- N店舗対応コストが O(N) → O(1)（プロンプトだけで新店舗に対応）
- UI変更耐性が抜群（セレクタの追従不要）
- データ取得から自動発注まで同じ技術スタックで拡張できる
- 2FA・CAPTCHA を自然に扱える
