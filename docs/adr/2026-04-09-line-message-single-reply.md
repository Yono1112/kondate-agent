# ADR: LINE 応答を「1ターン1メッセージ」に統一する

**日付**: 2026-04-09
**ステータス**: Accepted
**実装コミット**: `1776876`（`fix(agent): 重複メッセージ送信を防ぐため instruction の矛盾を解消`）

## 背景

`kondate-agent` の LINE 連携で、**1ターンのユーザー発言に対して2通のメッセージが届く** 現象が発生していた。

### 具体的な症状

ユーザー報告:
- 「夕食を提案して」→ ボタンテンプレのみ1通（問題なし）
- ボタン候補タップ → ボタンテンプレのみ1通（**料理詳細テキストが来るべきなのに来ない**）
- 「在庫確認したい」→ 在庫テキスト1通 **＋ 期限警告ボタンテンプレ1通（計2通）**

つまり「重複」と「取りこぼし」が場面によって入れ替わっていた。

## 根本原因

`src/mastra/agents/kondate-agent.ts` の instruction に **両立不可能な2つの指示** が同時に書かれていた。

### 矛盾①: テキスト空ルール

> send-line または send-line-buttons を呼び出した後は、テキスト応答を空文字列にする

### 矛盾②: ネクストアクション章

> 処理が完了したとき、ユーザーが次に何をしたいかを send-line-buttons で提示する
> - 消費期限チェック後 → text:「〇〇があと〇日です」 buttons:「使う献立を提案して」「わかった」
> - 食事記録後 → text:「記録しました！」 buttons: ...

**問題**: 「在庫確認 → 在庫リストのテキストを返す → さらに期限警告ネクストアクションボタンを出す」というフローでは **テキスト応答とボタンテンプレの2通送信が必要** なのに、別の指示では「ツール呼び出し後はテキスト空」と命じている。LLM はターンごとにどちらに従うかがブレる。

### 観察された挙動との整合

| 操作 | 結果 | 解釈 |
|---|---|---|
| 「夕食を提案して」 | ボタンテンプレのみ（1通） | LLMが「テキスト空」ルールを優先 |
| ボタン候補タップ | ボタンテンプレのみ（1通） | 同上。ただし**本来は料理詳細テキストを返すべき**（instruction 31行目）→ **取りこぼし** |
| 「在庫確認したい」 | 在庫テキスト + 期限警告ボタンテンプレ（2通） | LLMが「ネクストアクション」ルールを優先 |
| 食事記録・レシート解析 | 同様に2通 | ネクストアクション優先 |

つまり **「重複」に見えていた2通は LLM が仕様通り両方出していただけ**。別の場面で起きていた「詳細テキスト取りこぼし」も同じ矛盾の裏返し。

## 検討した選択肢

### 案A: 1通に統一する（シンプル優先）

- **やめること**: ネクストアクションボタンを毎回出す運用
- **ルール**: 「ツール呼び出し後はテキスト応答を空にする」を徹底し送信は1通のみ
- instruction からネクストアクション章を削除
- ボタンタップ後の料理詳細テキストは **webhook の `response.text` で返す**（send-line ツールを使わない）
- **UX**: スッキリするが、ユーザーが次のアクションを自分で打つ必要あり
- **実装**: instruction の書き換えのみ。コード変更なし

### 案B: ネクストアクション併用を正式仕様とする

- **やめること**: 「テキスト応答を空にする」という過剰な抑制
- **ルール**: 「メイン応答 + ネクストアクションボタン」の2通構成を正式に認める
- instruction から「テキスト応答を空にする」を削除
- 代わりに **献立提案シーンでは webhook 側でツール呼び出し検出→text抑制** をコードで担保
- **UX**: 操作を誘導できるが常に2通届く
- **実装**: instruction + webhook コードの改修

### 案C: ハイブリッド（場面ごとに明確化）

- 献立提案・ボタン選択後 → ボタンテンプレのみ
- 在庫確認・期限チェック・食事記録・レシート解析後 → 結果テキスト + ネクストアクションボタンの2通
- instruction を場面ごとに詳細化
- **UX**: 意図通り。ただし LLM の instruction 従順度に依存
- **実装**: instruction の記述力に全てを賭ける。最もブレやすい

## 決定

**案A（1ターン1メッセージ統一）を採用**。

## 理由

1. **LLM の instruction 従順度に依存せず、挙動が安定する**
   - 案C は「細かく書けば従ってくれるはず」という楽観論。GLM-5 系の従順度ではリスクが高い
   - 案A は「ルールが極端に単純」なので LLM が誤解する余地が少ない
2. **ユーザーが「重複」と感じている違和感を直接解消できる**
   - 2通届く体験そのものを排除する
3. **ボタンタップ後の詳細テキスト取りこぼしも自動解決**
   - テキストを返す唯一の経路が webhook `response.text` になるため、LLM が省略しづらい
4. **実装コストが最小**
   - instruction の書き換えのみ。webhook コードは触らない
5. **ユーザーの明示的な同意**
   - 3案提示してユーザーが案A を選択

## 却下した理由

- **案B**: 「2通届く」こと自体がユーザーの違和感の原因なので、正式仕様化は課題を解決しない
- **案C**: 場面ごとの例外が多く、LLM が誤解しやすい。矛盾の温床になりやすい

## 実装サマリ

`src/mastra/agents/kondate-agent.ts` の変更:

1. **「メッセージ送信ルール（最重要）」セクションを新設**
   - パターンA（テキスト応答のみ、ツール呼ばない）
   - パターンB（send-line-buttons + 空テキスト、献立提案のみ）
   - 禁止事項を明文化: 「1ターン中にボタン送信とテキスト応答の併用禁止」「ネクストアクション追加送信禁止」「send-line ツールを通常返信に使う禁止」

2. **「ネクストアクションの提示」章を削除**

3. **献立選択後の挙動を明確化**
   - ステップ5: 「send-line-buttons は使わず、テキスト応答として返す」

4. **send-line ツールの用途を限定**
   - 通常の返信では使わず、能動 push 通知専用

## 期待される挙動変化

| シーン | 変更前 | 変更後 |
|---|---|---|
| 夕食提案 | ボタンテンプレのみ（偶然OK） | ボタンテンプレのみ（安定） |
| ボタン候補タップ | ボタンテンプレのみ（バグ） | **料理詳細テキスト**（修正） |
| 在庫確認 | 在庫テキスト + 期限警告ボタン（2通） | **在庫テキストのみ**（修正） |
| 食事記録 | 記録テキスト + ネクストアクション（2通） | **記録テキストのみ**（修正） |
| レシート解析 | 追加通知 + ネクストアクション（2通） | **追加通知のみ**（修正） |

## リスクと追加対策

LLM は instruction 100% 遵守ではないため、案A でも重複が再発する可能性は残る。

**その場合の二段構え案**（未実装）:
- webhook ハンドラに「ツール呼び出しを検出したら `response.text` を強制無視する」ガードを追加
- コード側で LLM の暴走を吸収

現時点ではユーザーが実機で動作確認し「大丈夫そう」と判断したため、二段構えは実装しない。問題が再発したら追加する。
