# レシートOCR略称問題とネットスーパー経路の検討

**日付**: 2026-04-09
**関連エージェント**: kondate-agent（在庫管理）
**ステータス**: 調査完了、実装方針は未着手

## 背景・問題

既存の `parse-receipt` ツール（視覚LLMでレシート画像を解析）には **レシート印字の略称による誤認問題** がある。

**具体例**: サミットで購入した「たらこパスタソース」がレシート上では単に「たらこ」と印字されており、LLM が生たらこ（明太子的食材）と誤認して在庫登録してしまった。結果、下流の献立提案が破綻する。

## 第一原理分析

### 情報損失はどこで起きているか

POSは商品を **13桁のJANコード** で識別しているが、紙幅の都合でレシート印字時に文字列を切り詰めている。つまり「POS → 紙印字」段階の **不可逆変換** が根本原因。紙を眺めて精度を上げるのは理論上の上限がある。

### 取るべき戦略は2つに絞れる

1. **紙以外の信号を足す**
2. **紙を使うのをやめる**

紙レシート解析の磨き込みは本質的解決にならない。

### 真の情報ニーズ

エージェントが必要としているのは「正確な商品名」ではなく、**「今日何が作れるか」を判断できる食材メタデータ**:

- 食材カテゴリ（「パスタソース」か「生たらこ」かの区別がつく粒度）
- 量（g/個の概算）
- 消費期限（紙レシートには書かれていない → 別源が必要）
- 調理文脈（単体食材か、具材か、調味料か）

**重要な気付き**: 消費期限はもともと紙レシートに載っていない。つまり「紙のOCR完結」は最初から幻想。何らかの別信号は既に必要だった。

## 情報源の階層（正統性順）

1. **商品パッケージ本体**（JANバーコード、成分表、賞味期限印字）← 最も正統
2. **デジタルレシート / 電子レシート**（構造化JSON、JAN付き）
3. **スーパーのアプリ/ポイントカード履歴**
4. **紙レシートOCR** ← 現状
5. **ユーザー口述**
6. **AI推測**

紙レシートは下から2番目。

## 識別番号の正体: JAN vs PLU

ユーザーから「レシートに識別番号があるのでは」という指摘があった。調査結果:

| 種類 | 桁数 | 共通性 | 用途 |
|---|---|---|---|
| **JAN (Japan Article Number)** | 13桁 | **全国共通** | 商品パッケージのバーコード本体 |
| **PLU (Price Look-Up)** | 2〜6桁 | **チェーン独自** | POS内部の商品コード。レシートに印字されるのはこれが多い |
| **インストアコード** | 13桁（先頭"02"） | 店舗限定 | 生鮮・惣菜の店内ラベル |

**実態**: 日本のスーパーレシートで商品名の隣に付く短い数字はほぼ PLU。JANではない。JANなら楽天商品検索API / Yahoo!ショッピングAPI / Jancode Lookup で解決できたが、PLUはチェーン独自なので公開マスタはない。

**ただし PLU でも戦略的価値はある**: PLU は文字列略称より辞書キーとして優秀。

- 数字なのでOCR誤認が少ない
- チェーン内で一意
- 一度ユーザーが訂正すれば永続解決

→ 従来想定していた「文字列略称 → 正式名」辞書を、**「PLUコード → 正式名」辞書**に格上げする価値あり（ただしサミットのレシートに PLU が印字されているか要実物確認）。

## ネットスーパー経路（ユーザー提案、第一原理的に最強）

### なぜ強いか

| 項目 | 紙レシート | ネットスーパー |
|---|---|---|
| データ出所 | POS → 紙 → OCR → LLM | 注文データ（構造化JSON） |
| 商品名 | 略称で曖昧 | **正式名称フル** |
| 量 | 目視判定 | g/個/mL まで明記 |
| 価格 | OCR | 確定値 |
| 購入イベント | ユーザー撮影操作 | **配送時に自動発生** |
| 消費期限 | 不明 | カテゴリから推定しやすい |

**略称問題・識別番号問題・OCR誤認問題を全てまとめて解決する可能性**。

### 取得経路（ランキング）

#### 経路A: 注文確認メール → Gmail API でパース ← **最有力**

ほぼ全てのネットスーパーは注文確認メール・発送通知・配達完了メールを送る。本文にはテキストで商品明細が入る。Gmail 経由で取得・解析すれば:

- **完全自動**: ユーザー操作ゼロ
- **構造化**: メール本文は紙より遥かにパースしやすい
- **遡及可能**: 過去の注文メールから履歴再構築
- **Gmail MCP が既に本環境に存在** → 実装コスト低

実装スケッチ:
1. Mastra ワークフロー/スケジューラで定期的に Gmail を検索（`from:netsuper@summit.jp` 等）
2. 新着注文確認メールを LLM でパースして明細抽出
3. `manage-inventory` で自動在庫登録
4. LINE に「サミットネットスーパーから自動で7品追加しました」と通知

#### 経路B: Playwright によるスクレイピング

注文履歴ページにログイン。ただし2FA/CAPTCHA/規約違反リスクがあり、Gmail 経路があるなら冗長。

#### 経路C: 公式API

楽天西友ネットスーパーは楽天APIエコシステムに乗っている可能性。サミット・マルエツ・ライフの公式個人向けAPIは期待薄。

## サミット井土ヶ谷店レシート実物分析（2026-04-09）

ユーザーから 2026/04/07 のサミット井土ヶ谷店レシート写真を入手し分析した。

### レシート構造

```
サミット 井土ヶ谷店 ☎045-715-3310
登録番号 T3011301002747
2026年04月07日(火) 18:15 レジ0713

1100軽*  キャベツ           ¥158
1520軽*  ブルダック炒め麺     ¥238
1520軽*  コンソメ顆粒パウチ    ¥298
1520軽*  SB生風味たらこ      ¥218
  値引        20%          -¥44
1100軽*  くらしカットエリ     ¥128
1580軽*  国産丸大豆納豆      ¥128
1520軽*  結束スパ1.4        ¥328
  値引        20%          -¥66
合計                     ¥1,496
```

### 核心的発見: PLU は印字されていない

左端の4桁数字は **個別商品の PLU ではなく部門コード（カテゴリ）**。

| コード | 推定カテゴリ | 該当商品 |
|---|---|---|
| 1100 | 青果（野菜・きのこ） | キャベツ、くらしカットエリ |
| 1520 | 加工食品（乾物・調味料・麺） | ブルダック炒め麺、コンソメ、SB生風味たらこ、結束スパ |
| 1580 | 日配品（豆腐・納豆系） | 国産丸大豆納豆 |

`軽*` は軽減税率対象マーク（食品=8%）。

**結論: PLU ベースの辞書戦略は使えない。** 部門コードは「青果」「加工食品」程度の粒度しかなく、個別商品の特定には不十分。

ただし **部門コードはカテゴリ推論の補助として有効**:
- `1100` + 「たらこ」→ 青果売り場の生たらこ
- `1520` + 「たらこ」→ 加工食品のたらこソース

### 略称パターンの法則性

| レシート表記 | 正式名（推定） | 略称パターン |
|---|---|---|
| キャベツ | キャベツ | そのまま（短い商品名は無傷） |
| ブルダック炒め麺 | ブルダック炒め麺 | そのまま |
| コンソメ顆粒パウチ | 味の素コンソメ顆粒パウチ？ | ブランド名が脱落 |
| SB生風味たらこ | S&B 生風味たらこ（パスタソース） | ブランド名は残る、「パスタソース」が脱落 |
| くらしカットエリ | くらし良好 カットエリンギ | ブランド名切り詰め + 商品名切り詰め |
| 国産丸大豆納豆 | 国産丸大豆納豆 | ほぼそのまま |
| 結束スパ1.4 | 結束スパゲッティ 1.4mm | カテゴリ省略 + サイズは数字で残る |

**法則**:
1. ブランド名プレフィックスが残る傾向がある（"SB", "くらし"）
2. 短い商品名はそのまま印字される（キャベツ、納豆）
3. 長い商品名は末尾から切り詰められる（パスタソース→脱落、エリンギ→エリ、スパゲッティ→スパ）
4. サイズ・規格の数値は残る（1.4mm → 1.4）
5. 値引行は直前の商品に対する割引（20%オフ）

### 「SB生風味たらこ」問題の再評価

以前ユーザーが報告した「たらこパスタソースがたらこと誤認された」問題の実物を確認。実際のレシート表記は「SB生風味たらこ」で、以下の手がかりが存在する:
- "SB" = S&B食品（調味料・パスタソースメーカー）
- 部門コード "1520" = 加工食品（生鮮ではない）

つまりLLMに「SB = S&B食品」「部門コード1520 = 加工食品」という文脈を与えれば、「これはパスタソースであって生たらこではない」と推論できる材料は十分ある。

## 2026-04-09 追記: Computer Use を中核技術に決定

本調査では当初ネットスーパー経路の実装手段として Gmail API + Playwright MCP を想定していたが、ユーザーから **「Computer Use の精度がめちゃくちゃ高くなっているので、Computer Use で取った情報をそのまま在庫管理に転用できる」** という指摘があり、再分析した結果 **Computer Use を中核技術に据える** 方針に更新した。

### なぜ Computer Use が戦略的に正しいか

1. **N店舗対応コストが O(N) → O(1) に落ちる**
   - Playwright MCP だとサミット・マルエツ・ライフそれぞれに個別スクレイパーが必要で保守地獄
   - Computer Use なら「〇〇ネットスーパーで注文履歴を取って」のプロンプトだけで済む
   - 新店舗対応のコードコストがゼロに近い

2. **UI変更耐性が抜群**
   - 画素ベースで判断するので A/B テストや UI 刷新で壊れない
   - DOM セレクタの追従が不要

3. **データ取得から自動発注への拡張が連続的**
   - 同じ Computer Use エージェントが「履歴を読む」も「カートに入れて購入確定」もできる
   - Playwright/Gmail 経路だと自動発注フェーズで別の大掛かりな実装が必要
   - **ユーザーが memory に保存した「究極ビジョン（完全自律購買）」への唯一の自然なパス**

4. **メールでは取れない情報にアクセスできる**
   - セール、クーポン、代替品情報、商品詳細、価格変動
   - これらは能動的な献立提案の材料になる（「明日サミットでキャベツ¥98だから麻婆キャベツ作る？」）

5. **モバイルアプリ専用サービスにも対応できる**
   - Oisix 等はアプリ中心
   - Android エミュレータを Computer Use で操作する道が開ける

6. **2FA・CAPTCHA を自然に扱える**
   - Playwright では対応が面倒だが、Computer Use は「OTP 入力画面が出たら LINE でユーザーに確認」という流れが自然に書ける

### Computer Use vs Gmail vs Playwright MCP 比較表

| 軸 | Gmail | Playwright MCP | Computer Use |
|---|---|---|---|
| 初期コスト | 低 | 中 | 高 |
| 継続コスト | 低 | 低 | 中〜高 |
| UI変化耐性 | 高（メール） | 低 | 最高 |
| 複数店舗対応 | ◎ | △（店舗ごとに実装） | ◎ |
| 2FA対応 | ◎ | × | ○ |
| データ完全性 | △（メール次第、代替品反映されず） | ○ | ◎ |
| 自動購買への拡張性 | × | ○ | ◎ |
| 運用複雑度 | 低 | 中 | 高 |
| モバイル専用アプリ | × | ×（Appium別途必要） | ○（エミュレータ操作） |

### 各経路の役割分担（結論）

- **Gmail 経路**: 主役ではなく、軽量トリガーとして共存
  - 配達完了メール検知 → Mastra ワークフロー起動 → Computer Use を呼び出し
  - これにより Computer Use を常時動かさず済む（コスト最適化）
- **Computer Use**: 中核技術
  - 注文履歴抽出、商品詳細取得、セール情報収集、将来的な自動発注
- **Playwright MCP**: kondate-agent 本番フローからは外す
  - Computer Use で代替可能
  - kondate-agent 用途では不要になる見込み

### Computer Use 採用に伴う懸念と対策

| 懸念 | 対策 |
|---|---|
| コスト（1アクション数秒・数セント） | Gmail トリガーで起動頻度を絞る、結果キャッシュ、差分更新 |
| 認証情報の保管 | OS キーチェーン or KMS + 暗号化 env、**devcontainer には一切持ち込まない**（git identity leak の教訓） |
| サンドボックス運用 | Anthropic 公式 Docker reference impl ベース、ネットワーク egress を ACL で絞る（init-firewall.sh 流用）、画面に機密情報が映らないようにする |
| 非決定性（誤クリック） | **書き込み系操作は必ず LINE 承認ゲート**、操作ログ全保存でポストモーテム可能に |
| 利用規約 | 各店舗の規約確認、個人自己利用の範囲に留める、レート制限遵守 |
| 成熟度 | Phase 1 は読み取り専用、書き込み系（Phase 4）は精度が安定してから、常に人間監視を残す |

### 実装ロードマップ（更新版）

#### Phase 1: Computer Use PoC（データ取得、手動トリガー）
- Anthropic Computer Use reference impl（Docker / Xvfb + Firefox）をベースにセットアップ
- 認証情報は暗号化保管
- LINE 発言「注文履歴を取って」で Mastra ワークフローから Computer Use 起動
- 結果を LINE に要約表示 → ユーザー承認で DB 書き込み
- 検証ポイント: ログインフロー、抽出精度、コスト、所要時間、UI変更耐性

#### Phase 2: Gmail トリガー化
- 配達完了メール検知 → 自動で Phase 1 パイプライン起動
- ユーザー操作ゼロ化
- 紙レシート経路と並行運用

#### Phase 3: 店舗拡大
- マルエツ・ライフ・Oisix 等へ横展開
- Computer Use の強みで実装コストほぼゼロ

#### Phase 4: 自動発注（究極ビジョン到達）
- 在庫 + 献立 + 予算から不足品リスト生成 → Computer Use でカート投入
- 決済前に LINE 承認ボタン → 承認後に購入確定
- 金額上限・週次予算・ホワイトリスト・高額品は必ず人間承認

### 本追記による当初記述の位置づけ変更

本ドキュメント前半の「ネットスーパー経路 > 経路A: Gmail API」はまだ有効だが、**最終形態ではなく Phase 2 の軽量トリガー**という位置づけに変更。「経路B: Playwright」は kondate-agent 本番フローからは外し、Computer Use に一本化。

当初の「紙レシート + PLU 学習層」も引き続き並行運用（Phase 2 以降もサミット実店舗の紙レシート経路はカバー対象）だが、長期的にはネットスーパー利用比率を上げて紙レシート依存を減らす方針。
