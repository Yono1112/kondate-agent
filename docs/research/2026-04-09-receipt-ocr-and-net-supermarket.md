# レシートOCR略称問題とネットスーパー経路の検討

**日付**: 2026-04-09
**関連エージェント**: kondate-agent（在庫管理）
**ステータス**: 調査完了、実装方針は未着手

## 背景・問題

既存の `parse-receipt` ツール（視覚LLMでレシート画像を解析）には **レシート印字の略称による誤認問題** がある。

**具体例**: サミットで購入した「たらこパスタソース」がレシート上では単に「たらこ」と印字されており、LLM が生たらこ（明太子的食材）と誤認して在庫登録してしまった。結果、下流の献立提案が破綻する。

## 第一原理分析

### 情報損失はどこで起きているか

POSは商品を **13桁のJANコード** で識別しているが、紙幅の都合でレシート印字時に文字列を切り詰めている。つまり「POS → 紙印字」段階の **不可逆変換** が根本原因。紙を眺めて精度を上げるのは理論上の上限がある。

### 取るべき戦略は2つに絞れる

1. **紙以外の信号を足す**
2. **紙を使うのをやめる**

紙レシート解析の磨き込みは本質的解決にならない。

### 真の情報ニーズ

エージェントが必要としているのは「正確な商品名」ではなく、**「今日何が作れるか」を判断できる食材メタデータ**:

- 食材カテゴリ（「パスタソース」か「生たらこ」かの区別がつく粒度）
- 量（g/個の概算）
- 消費期限（紙レシートには書かれていない → 別源が必要）
- 調理文脈（単体食材か、具材か、調味料か）

**重要な気付き**: 消費期限はもともと紙レシートに載っていない。つまり「紙のOCR完結」は最初から幻想。何らかの別信号は既に必要だった。

## 情報源の階層（正統性順）

1. **商品パッケージ本体**（JANバーコード、成分表、賞味期限印字）← 最も正統
2. **デジタルレシート / 電子レシート**（構造化JSON、JAN付き）
3. **スーパーのアプリ/ポイントカード履歴**
4. **紙レシートOCR** ← 現状
5. **ユーザー口述**
6. **AI推測**

紙レシートは下から2番目。

## 識別番号の正体: JAN vs PLU

ユーザーから「レシートに識別番号があるのでは」という指摘があった。調査結果:

| 種類 | 桁数 | 共通性 | 用途 |
|---|---|---|---|
| **JAN (Japan Article Number)** | 13桁 | **全国共通** | 商品パッケージのバーコード本体 |
| **PLU (Price Look-Up)** | 2〜6桁 | **チェーン独自** | POS内部の商品コード。レシートに印字されるのはこれが多い |
| **インストアコード** | 13桁（先頭"02"） | 店舗限定 | 生鮮・惣菜の店内ラベル |

**実態**: 日本のスーパーレシートで商品名の隣に付く短い数字はほぼ PLU。JANではない。JANなら楽天商品検索API / Yahoo!ショッピングAPI / Jancode Lookup で解決できたが、PLUはチェーン独自なので公開マスタはない。

**ただし PLU でも戦略的価値はある**: PLU は文字列略称より辞書キーとして優秀。

- 数字なのでOCR誤認が少ない
- チェーン内で一意
- 一度ユーザーが訂正すれば永続解決

→ 従来想定していた「文字列略称 → 正式名」辞書を、**「PLUコード → 正式名」辞書**に格上げする価値あり（ただしサミットのレシートに PLU が印字されているか要実物確認）。

## スーパー別 電子レシート対応状況調査

ユーザーの買い物頻度: **サミット > マルエツ > ライフ**

### ライフ: ✅ 対応確定

- **2026年2月19日にライフ公式アプリがリニューアル**
- 東芝テックの **スマートレシート®** 連携で電子レシート機能を追加
- PayPay / d払い / au PAY / 楽天ペイ連携も追加
- 2026年3月1日〜4月30日にキャンペーン実施中（新規連携ユーザー1000人に1000ポイント）
- Source: [ライフ公式アプリリニューアル発表 (NEWSCAST)](https://newscast.jp/news/9132707)

### サミット: ❓ 対応不明、おそらく非対応

- サミット公式アプリ（ポイント・電子マネー用途）にはスマートレシート機能が統合されていない
- スマートレシート加盟店一覧に明示的な掲載が見つからない
- 東急ストア・マキヤ等は明示的に加盟店と記載されている
- **判断保留**: ユーザーに実際のスマートレシートアプリで井土ヶ谷店を検索してもらう必要あり（まだ実施していない）
- Source: [サミット公式アプリ案内](https://www.summitstore.co.jp/app-about/)、[スマートレシート使えるお店（公式）](https://www.smartreceipt.jp/store.html)

### マルエツ: ❌ 情報なし、対応していない可能性高い

- スマートレシート加盟店としての記載なし
- マルエツ公式アプリにも電子レシート機能は確認できない

### スマートレシート個人向けAPIの有無

**結論: 個人開発者が自分のデータを取得する正規ルートは存在しない**

- 東芝テック/東芝データが提供しているAPIは B2B（加盟店企業・購買統計データ提供）専用
- コープこうべFAQに「スマートレシートの情報をCSV/TSV等のテキスト形式で出力することはできません」と明記
- つまり仮にサミットが対応していても、**プログラムから自動取得する正規ルートはない**
- Source: [東芝データ 購買統計API (東芝データ)](https://www.global.toshiba/jp/news/data-corp/2021/05/20210511.html)、[コープこうべ スマートレシートFAQ](https://faq.coop-kobe.net/faq_detail.html?id=3019)

## 解決戦略の全網羅

### 象限A: 別の信号源に切り替える

| 案 | 精度 | UX摩擦 | 実装コスト |
|---|---|---|---|
| A1. 商品パッケージの個別撮影 | 高 | 高（1品1枚） | 低 |
| A2. 買い物袋ドサ撮り（一括撮影） | 中〜高 | 低（1枚） | 中 |
| A3. JANバーコード連続スキャン | 最高 | 中（ピッピッ） | 中 |
| A4. 電子レシートサービス連携 | 最高 | 低 | 中〜高 |
| A5. 家計簿アプリAPI流用（Dr.Wallet 等） | 中 | 低 | 高 |

### 象限B: 紙レシートを強化する

| 案 | 精度 | UX摩擦 | 実装コスト |
|---|---|---|---|
| B1. 店舗別「略称→正式名」辞書を学習 | 中（漸近向上） | 低 | 低 |
| B2. 曖昧項目のみLINE Quick Replyで確認 | 高 | 中 | 低 |
| B3. 価格＋カテゴリ＋店舗から文脈推論 | 中 | 低 | 中 |
| B4. 紙レシート + 買い物袋写真の2枚使い | 高 | 低 | 低 |

### 象限C: 精度を諦めて下流で補償

- C1. 低信頼度アイテムを「未確定」状態で在庫登録し、使う直前に確認
- C2. 自動登録を「下書き」扱いにし、1タップ承認UI

### 象限D: プロダクト境界を引き直す

- D1. レシート機能を捨てて、音声/テキスト入力に特化
- D2. 在庫管理自体を捨てて、冷蔵庫写真ベースに変える

## ネットスーパー経路（ユーザー提案、第一原理的に最強）

### なぜ強いか

| 項目 | 紙レシート | ネットスーパー |
|---|---|---|
| データ出所 | POS → 紙 → OCR → LLM | 注文データ（構造化JSON） |
| 商品名 | 略称で曖昧 | **正式名称フル** |
| 量 | 目視判定 | g/個/mL まで明記 |
| 価格 | OCR | 確定値 |
| 購入イベント | ユーザー撮影操作 | **配送時に自動発生** |
| 消費期限 | 不明 | カテゴリから推定しやすい |

**略称問題・識別番号問題・OCR誤認問題を全てまとめて解決する可能性**。

### 取得経路（ランキング）

#### 経路A: 注文確認メール → Gmail API でパース ← **最有力**

ほぼ全てのネットスーパーは注文確認メール・発送通知・配達完了メールを送る。本文にはテキストで商品明細が入る。Gmail 経由で取得・解析すれば:

- **完全自動**: ユーザー操作ゼロ
- **構造化**: メール本文は紙より遥かにパースしやすい
- **遡及可能**: 過去の注文メールから履歴再構築
- **Gmail MCP が既に本環境に存在** → 実装コスト低

実装スケッチ:
1. Mastra ワークフロー/スケジューラで定期的に Gmail を検索（`from:netsuper@summit.jp` 等）
2. 新着注文確認メールを LLM でパースして明細抽出
3. `manage-inventory` で自動在庫登録
4. LINE に「サミットネットスーパーから自動で7品追加しました」と通知

#### 経路B: Playwright によるスクレイピング

注文履歴ページにログイン。ただし2FA/CAPTCHA/規約違反リスクがあり、Gmail 経路があるなら冗長。

#### 経路C: 公式API

楽天西友ネットスーパーは楽天APIエコシステムに乗っている可能性。サミット・マルエツ・ライフの公式個人向けAPIは期待薄。

## ユーザーが提示した究極ビジョン

**「AIエージェント自身がネットスーパーから商品を自動発注する」**

詳細は `memory/vision_full_autonomy.md` に保存済み。在庫→献立→発注の一気通貫自動化が最終ゴール。

- Playwright MCP が本環境に既にあるので試作可能
- 承認フローは `send-line-buttons` の既存基盤で実装できる
- セキュリティ対策: 金額上限、週次予算、ホワイトリスト、高額商品は人間承認必須

## ROI 評価と推奨戦略

### ROI順

1. **A2 (買い物袋ドサ撮り) + B4 (紙と併用)** — 最推奨の現実解
2. **B1 (店舗別辞書) + B2 (曖昧項目確認)** — 即実装可能な追加層
3. **A3 (JANバーコード)** — 精度最高だがUX重い
4. **A4 (電子レシート連携) = ライフのみ** — 対応店に限り勝ち確定
5. **ネットスーパー Gmail 経路** — 中長期の本命

### ハイブリッド運用の提案

| 購入タイプ | 経路 | データ品質 |
|---|---|---|
| 定番（米・水・調味料・冷凍） | サミットネットスーパー → Gmail自動取得 | 最高 |
| 日常の生鮮・特売 | サミット実店舗 → 紙レシート + PLU学習 | 中 |
| 緊急買い足し | マルエツ/ライフ実店舗 → 紙レシート | 中〜低 |
| ライフで計画買い | ライフ店舗 → スマートレシートアプリ → スクショ | 高 |

重い米・水・飲料をネットスーパーに切り替えるだけで物理的負担も減るという副次効果もある。

### 3層防御（実装ロードマップ案）

#### 第1層: 入力経路を複線化
- レシート写真（現状維持、価格・日付・店舗の権威ソース）
- 買い物袋ドサ撮り（新規・商品内容の権威ソース）
- テキスト入力（フォールバック）
- JANスキャンは将来オプション

#### 第2層: 紙レシート解析を「下書き生成器」に再定義
- 解析結果を即登録せず confidence 付き下書き化
- 低信頼度アイテムだけ LINE Quick Reply で確認
- 訂正を `store_aliases` テーブル（または `store_plu_dictionary`）に学習保存
- 次回同店舗では辞書をプロンプトに注入

#### 第3層: 電子レシート/ネットスーパー対応
- ライフ: スマートレシートアプリのスクショを parse-receipt で流用
- サミット: まず対応可否を確認、対応していれば同じくスクショ経路
- サミットネットスーパー: Gmail 注文確認メールパイプライン構築

## 未着手の確認事項

1. **サミット井土ヶ谷店のスマートレシート対応可否** — ユーザーが実機で確認する必要あり
2. **サミットのレシート実物の PLU 印字有無** — ユーザーが次回買い物時にレシート写真を送る予定
3. **サミットネットスーパー注文確認メールのフォーマット** — 過去に利用していれば Gmail で確認可能

## 決定事項

- 本調査の結果、**紙レシート完結の解決策には理論上限がある**という第一原理的な結論に到達
- 短期: PLU ベース辞書学習層（サミット向け、レシート実物確認後）
- 中期: Gmail 経路でネットスーパー連携（サミット主軸）
- 長期: エージェント自動発注（memory/vision_full_autonomy.md 参照）
